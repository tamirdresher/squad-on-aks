# KEDA Autoscaling for Squad

Scale AI agent pods based on workload using [KEDA](https://keda.sh) (Kubernetes Event-Driven Autoscaling).

## Overview

Squad workloads are bursty — zero activity most of the time, then bursts when issues are created. KEDA enables **scale-to-zero** (save money) and **scale-on-demand** (handle bursts).

## Tier 1: GitHub Runner Scaler (Built-in)

KEDA has a native `github-runner` scaler that monitors pending GitHub Actions workflow jobs.

**Key finding:** This scaler already reads `X-RateLimit-Remaining` and `X-RateLimit-Reset` headers, implementing automatic backoff when rate-limited.

```yaml
# In values.yaml
keda:
  enabled: true
  picard:
    minReplicaCount: 0
    maxReplicaCount: 3

# ScaledObject configuration (auto-generated by Helm chart)
triggers:
  - type: github-runner
    metadata:
      owner: "your-org"
      runnerScope: "repo"
      repos: "your-repo"
      targetWorkflowQueueLength: "1"
      enableEtags: "true"  # Reduces API consumption via HTTP 304
```

**Effort:** Configuration only — no custom code.

## Tier 2: Metrics-API Scaler + Custom Adapter

For scaling based on GitHub API rate limits directly (not just runner queue length):

1. Build a lightweight HTTP adapter (~30 lines):
```python
# github_metrics_adapter.py
from flask import Flask, jsonify
import requests, os

app = Flask(__name__)

@app.route('/metrics')
def metrics():
    r = requests.get('https://api.github.com/rate_limit',
        headers={'Authorization': f'token {os.environ["GH_TOKEN"]}'})
    data = r.json()
    return jsonify({
        'remaining': data['rate']['remaining'],
        'limit': data['rate']['limit'],
        'reset': data['rate']['reset']
    })
```

2. Deploy the adapter as a small pod in the squad namespace.

3. Configure KEDA:
```yaml
triggers:
  - type: metrics-api
    metadata:
      url: "http://github-metrics-adapter.squad.svc/metrics"
      valueLocation: "remaining"
      targetValue: "100"  # Scale up when rate limit drops below 100
      format: "json"
```

**Effort:** ~30 lines of code + deployment manifest.

## Tier 3: Custom External Scaler (Open Source Opportunity)

For Copilot token budget scaling — no existing scaler handles this.

**Why custom?** Copilot token limits are:
- Organization-level billing data (not in HTTP headers)
- Requires GitHub GraphQL API
- Not exposed by any existing KEDA scaler

**Open source plan:** Build `keda-github-scaler` as a standalone repo:
- gRPC external scaler implementing KEDA's `externalscaler.proto`
- Scale on: Copilot token budget remaining, API rate limits, issue queue depth
- License: MIT

See: [KEDA External Scaler spec](https://keda.sh/docs/latest/concepts/external-scalers/)

## Comparison

| Approach | Scale Trigger | Effort | Production Ready |
|----------|--------------|--------|-----------------|
| github-runner | Workflow job queue | Config only | ✅ Yes |
| metrics-api + adapter | API rate limit remaining | ~30 LOC | ✅ Yes |
| Custom external scaler | Copilot tokens, custom | New project | ⚠️ Building |

## Setup

### 1. Enable KEDA on your AKS cluster

```bash
az aks update --resource-group $RG --name $CLUSTER --enable-keda
```

### 2. Enable in Helm values

```yaml
# values.yaml or --set
keda:
  enabled: true
  picard:
    minReplicaCount: 0
    maxReplicaCount: 5
    pollingInterval: 60
    cooldownPeriod: 300
```

### 3. Deploy

```bash
helm upgrade squad-agents ./helm/squad-agents --set keda.enabled=true
```

### 4. Verify

```bash
kubectl get scaledobject -n squad
kubectl get hpa -n squad  # KEDA creates an HPA
```
